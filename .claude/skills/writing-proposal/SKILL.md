# Proposal Writing Standards

## 情報収集（執筆前に必ず確認）

以下が不足している場合はユーザーに質問する。推測で埋めない。

1. **誰に**: 決裁者・提案先（役職・権限レベル）
2. **何を**: 提案内容の一文要約
3. **なぜ今**: 提案のトリガーとなった事実・データ
4. **いくら**: 費用（新規追加額 + 総額の両方）
5. **既決事項**: 過去に承認済みの関連決裁・前提条件
6. **媒体**: Slack / メール / ドキュメントのいずれか

## 構成テンプレート

以下の順序で構成する。テンプレートに従えば PREP法・SCQ・ピラミッド構造が自動的に適用される。

```
【依頼者】名前
【内容】提案の一文要約（= 結論ファースト）

【目的】
現状の説明（Situation）→ 問題・変化（Complication）→ 提案の方向性

■ 前提
• 既決事項・合意済みの条件（差分の基準点を明示）
• 現在の数字・ファクト

■ 提案
• 具体的なアクション（対象をセグメント分けして段階的に提示）
    ◦ 根拠となる数字・比較
    ◦ 確度ラベル（「体感」「推定」「公式発表」等）
• 承認後のアクション宣言

【費用】新規追加額（総額）/ 単位
【支払方法】支払手段
```

### セクション別の注意点

- **【内容】**: 1行で判断材料の全体像がわかること
- **【目的】**: 「〜したい」で終わる。判断の背景を3行以内で
- **■ 前提**: 「前回の決裁」や「既存の契約」など、今回の提案が差分であることを示す基準点を必ず置く
- **■ 提案**: 対象をティアリング（セグメント分け）し、それぞれに最適な施策を提示する。投資は段階的に検証すべきであり、提案の構造もそれを反映する
- **【費用】**: 新規追加額と総額の二段表示を必須とする

## 執筆原則

### 1. 数字で語る
- 抽象的な表現（「大幅に増加」「かなりのコスト」）は禁止。必ず定量化する
- 消費速度・内訳・コスト比較など、判断に必要な数字を揃える

### 2. 既決事項からの差分で提示する
- ゼロから説明しない。「前回〇〇が承認済み。今回の変更点は〜」の形にする
- 決裁者が「何が新しい判断か」だけを読めば済むようにする

### 3. 確度を正直に表示する
- 公式情報 → そのまま記載
- 体感・推定 → 「体感では」「推定」と明記
- 未確認 → 「未確認だが」と前置きする

### 4. 承認後のアクションを宣言する
- 提案の末尾に「承認後に何をするか」を書く
- 決裁者が承認ボタンを押した後の世界を想像できるようにする

## 媒体別の調整

| 媒体 | 調整内容 |
|------|----------|
| Slack | 上記テンプレートをそのまま使用。mrkdwn 記法で装飾。`/writing-slack-message` のルールに従う |
| メール | 件名に【決裁依頼】or【ご提案】を付与。本文は同じ構成 |
| ドキュメント | 見出しレベルを使って階層化。補足資料はリンクで添付 |

## AI臭さの除去（humanizer）

提案テキストを作成したら、送信前に `/humanizer` スキルを適用してAI臭さを除去する。

提案テキストで特に注意すべきパターン:

- 「〜することで、〜を実現します」のような抽象的な効果の羅列を避ける
- 箇条書きの各項目を同じ構文で揃えすぎない（文体の揺らぎを自然に残す）
- 「つきましては」「何卒」等の過剰な敬語の積み上げを避ける
- 提案理由を美化しない。事実と数字で淡々と書く

## 送信前チェックリスト

1. 【内容】を読むだけで提案の全体像がわかるか
2. 既決事項（前回の承認内容）が明記されているか
3. 数字が入っているか（費用・実績・比較）
4. 費用は新規追加額と総額の両方が記載されているか
5. 確度が不明な情報に「体感」「推定」等のラベルがあるか
6. 承認後のアクションが書かれているか
7. 段階的な検証ができる構造になっているか（セグメント分け・ステップ設計）
8. AI臭い表現が残っていないか（上記 humanizer セクション参照）

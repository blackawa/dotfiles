# clasp プロジェクト デプロイ戦略

GASを本番運用に出すための実装パターンと判断軸。
前提セットアップは `setup-clasp-project`、テストは `test-clasp-project` を参照。

## デプロイの全体像

GAS のデプロイには複数の意味がある:

| 用語 | 意味 | 使う時 |
|------|------|--------|
| **`clasp push`** | ローカルのコードをスクリプトに反映する | 開発中の更新 |
| **API実行可能デプロイ** | `clasp run` を可能にする内部デプロイ | テスト用、1回作れば触らない |
| **本番リリース** | 別のスプシに同じコードを移植する | 開発スプシ→本番スプシへの移行 |

このスキルは主に **本番リリース** を扱う。

## 実行権限の選択肢

GASは「誰のアカウントで実行されるか」が運用上の重大ポイント。先に決めること。

| パターン | 実行アカウント | 利点 | 欠点 |
|---------|--------------|------|------|
| **開発者がスプシのGAS編集権限を持つ** | 開発者 | clasp push で直接デプロイ可能 | 開発者退場時に切替が必要 |
| **エンドユーザーが自分でアップロード** | エンドユーザー | 退場の影響なし、認可も自身 | 手作業でファイル受け渡し |
| **サービスアカウント** | SA | 安定・自動化向き | スプシ・BQ両方に権限付与が必要、複雑 |

開発者にGAS編集権限が下りない場合（admin制限など）、`エンドユーザーが自分でアップロード` パターンが現実解。本スキルはそのケースも含めて扱う。

## 環境別設定の管理

開発スプシと本番スプシで `SPREADSHEET_ID` などの定数が異なるので、設定をどう切り替えるかを設計する。

### 推奨: 設定ファイルにハードコード + デプロイ時に書き換え

```javascript
// プロジェクト_設定.js
const CONFIG = {
  BQ_PROJECT: 'project-id',
  // 本番デプロイ時に本番スプシIDへ書き換える（手順書参照）
  // 本番: '<PROD_ID>'
  // 開発: '<DEV_ID>'
  SPREADSHEET_ID: '<DEV_ID>',
  SHEET_MARKER: '★',
};
```

利点:
- コードを読んで「本番値はどれか」がコメントで分かる
- リリース用バンドル生成スクリプトで自動置換できる
- PropertiesService等の動的取得を使うより、コードレビューしやすい

### 代替案: PropertiesService 経由

```javascript
const SPREADSHEET_ID = PropertiesService.getScriptProperties().getProperty('SPREADSHEET_ID');
```

- 利点: コード変更なしに本番／開発を切り替えられる
- 欠点: 初回セットアップで Properties をGUIから設定する必要がある
- エンドユーザーがセットアップする場合、手順が増えるので非推奨

## ファイル分離の方針（本番に持ち込まない）

`プロジェクト_テスト.js` や `プロジェクト_dev_*` 関数は本番に持ち込まない。
ファイル単位で分離していれば、リリース時にコピーから外すだけで済む。

| ファイル | 本番デプロイ | 理由 |
|---------|------------|------|
| `appsscript.json` | ✅ | API有効化等の設定 |
| `プロジェクト_設定.js` | ✅（IDを書き換え後） | 全ファイルから参照される定数 |
| `プロジェクト_メイン.js` | ✅ | 接続確認・本番ヘルパー |
| `プロジェクト_バッチ.js` 等 | ✅ | 本番ロジック |
| `プロジェクト_テスト.js` | ❌ | 開発用、`dev_` 関数も含む |
| `.clasp.json` | ❌ | 開発者の clasp 設定 |
| `.clasprc.json` | ❌ | 認証情報（秘密） |

## 別アカウントでのアップロード手順（エンドユーザーが実施）

開発者にGAS編集権限がない場合のリリース手順。

### 1. 開発者がファイルを準備

- 本番用バンドルとして本番に持ち込むファイルだけを zip
- 設定ファイルは本番値に書き換え済み
- README.txt を添付（用途と注意点）

### 2. エンドユーザーが Apps Script エディタで作業

1. 本番スプシで「拡張機能 → Apps Script」
2. ファイルを1つずつ作成して中身を貼り付け
   - **設定ファイルから先に作成する**（他ファイルから参照される）
3. `appsscript.json` も忘れず更新（左下「プロジェクトの設定」→「マニフェストを表示」にチェック）
4. 保存

### 3. 初回認証フロー（重要）

ユーザーアカウント名義での認可が必要。

1. エディタで簡単な関数（例: `プロジェクト_ping`）を実行
2. 認証ダイアログ → 「権限を確認」 → ユーザーアカウントを選択
3. 「詳細」→「[プロジェクト名] (安全ではないページ) に移動」
4. 「許可」

**以後、夜間バッチ等もユーザーアカウント名義で実行される**。BigQuery等の他リソースへの権限もユーザーに必要。

### 4. リリース立ち会いは Meet で

- ユーザーが操作する画面を共有してもらい、開発者が手順書を読みながら案内する
- 認証や権限で詰まる可能性があるため対面（オンライン）で実施するのが堅実

## トリガーの設定

時間ベースのトリガーは GUI から設定するのが堅実（`ScriptApp.newTrigger` で動的設定もできるが、運用者が後から確認・変更できる方が安全）。

### 手動設定の手順

1. Apps Script エディタの左サイドバー「トリガー」（時計アイコン）
2. 「+ トリガーを追加」
3. 設定:
   - 実行する関数: `プロジェクト_nightlyBatch`
   - イベントのソース: **時間主導型**
   - タイプ: **日付ベースのタイマー**
   - 時刻: 業務に合わせる
4. 保存（認証ダイアログが出たら許可）

### 時刻の考慮

| 実行時刻帯 | 利点 | 欠点 |
|----------|------|------|
| 早朝（午前2〜5時） | 業務時間と競合しない、夜間バッチ向け | 上流データ（API取得等）が間に合っていない可能性 |
| 業務開始前（午前7〜8時） | 上流データが揃った状態で動ける | 実行時にユーザー編集中のリスク |
| 業務終了後（午後10時〜） | 当日データが揃う | 翌朝までにリトライ余地がない |

上流データ（DataBeat等）の取得完了タイミングを確認してから決める。

## 「緊急ボタン」「実行UI」のデプロイ

スプシ上のボタン（図形）にスクリプトを割り当てる場合、**ボタンの実体はスプシのオブジェクト**でGASのコードではない。

- 開発スプシで作ったボタンを本番にコピーするには「シートごと複製」
- またはエンドユーザーに本番スプシで新規ボタンを置いてもらい、関数名だけ伝える
- 配置場所は本番運用者の都合に委ねるのがよい（受入テストで決める等）

## 本番後の事故からの復旧

### コードのロールバック

`clasp push` で過去版を上書きしただけでは履歴に残らないので、**git で必ずコミット**してから push する習慣を付ける。

```bash
git add gas/ && git commit -m "本番デプロイ前の状態" && clasp push
```

事故が起きたら:

```bash
git revert <commit>
clasp push
```

### スプシの巻き戻し

スプシ自体に書き込みエラーが起きた場合:
- スプシの「変更履歴」から特定時点の状態に戻せる（ファイル → バージョン履歴）
- ただしテーブル全体の戻しになるので慎重に

## 受け渡し方法の選択

| 方法 | 利点 | 欠点 |
|------|------|------|
| **チャットで zip 添付** | 手軽、履歴に残る | バージョンが分散しがち |
| **Drive 共有** | 最新版を更新可能 | 共有設定の手間、誤削除リスク |
| **clasp 連携（権限あり）** | 最新版が常に同期 | エンドユーザーが clasp を使えること |

エンドユーザーが受け取って自分でアップロードするケースでは「zip 添付＋ README」が最もシンプル。
